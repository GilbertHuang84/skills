# Resume Optimizer — 简历优化系统

## 设计哲学

**核心原则：简历不是一个"自我表达"的文档，而是一个"让不同角色的人在不同目的下都能快速找到他们要的信息"的文档。**

**内外双轨体系：**
- **内部简历**：个人档案馆，忠实记录所有经历，主要读者是自己（由 resume-analyzer 维护）
- **外部简历**：针对特定目标定制，主要读者是企业方（由本 SKILL 生成）

**本 SKILL 的定位：**
- 本 SKILL 是一个**生成器**，不负责信息收集和简历分析
- 信息收集和四维审视由 [resume-analyzer](../resume-analyzer/SKILL.md) 完成
- 本 SKILL 基于 resume-analyzer 的审阅结果，为特定目标岗位定制外部简历
- 生成过程中发现的新认知，通过回写机制同步更新内部简历和用户画像

## 输入来源

| 来源 | 提供的信息 | 精准度 |
|------|-----------|--------|
| **resume-analyzer 审阅报告** | 信息提取结果 + 四维审视结果 + 发现的问题 + 建议 | 最高 |
| **internal-resume.md** | 完整的经历档案（由 resume-analyzer 维护） | 最高 |
| **profile.md** | 用户画像、能力标签、成长轨迹 | 高 |

**输入文件路径：**
- `data/{用户标识}/resume-reviews/review-{日期}.md` — 审阅报告
- `data/{用户标识}/internal-resume.md` — 内部简历
- `data/{用户标识}/profile.md` — 用户画像

**优先使用最新的审阅报告。** 如果有多份审阅报告，按日期倒序取最新的。

## 存档协议

存档文件路径统一指向 `data/{用户标识}/` 目录：

```
data/
└── {用户标识}/
    ├── internal-resume.md              # 内部简历（由 resume-analyzer 维护，本 SKILL 可回写）
    ├── profile.md                      # 用户画像（本 SKILL 可回写）
    ├── external-resumes/               # 外部简历（按目标分别存储，本 SKILL 产出）
    │   ├── resume-{目标1}.md
    │   └── resume-{目标2}.md
    └── resume-reviews/                 # 审阅报告（由 resume-analyzer 产出，本 SKILL 读取）
        └── review-{日期}.md
```

存档模板见 [references/archiving-protocol.md](references/archiving-protocol.md)。

## Phase 1：读取审阅结果

**目标：** 从 resume-analyzer 的审阅报告中获取完整的分析结果，作为生成外部简历的输入。

### Step 1.1 检测审阅报告

检查 `data/{用户标识}/resume-reviews/` 下是否存在审阅报告：

- **有审阅报告** → 读取最新的一份，进入 Step 1.2
- **无审阅报告** → 提示用户先运行 resume-analyzer，或进入预览模式（见下方"预览模式"说明）

### Step 1.2 读取审阅内容

从审阅报告中提取以下信息：

| 审阅报告章节 | 提取内容 | 用途 |
|-------------|---------|------|
| 经历清单 | 所有经历的时间线、类型、完整度 | 了解用户经历全貌 |
| 信息提取 | 核心贡献、能力证据、成长轨迹、被忽略亮点、可迁移能力、量化数据 | 生成外部简历的素材库 |
| 四维审视结果 | 各维度的判定（✅/⚠️/❌）、关键发现、红灯信号 | 确保外部简历在每个维度都有证据 |
| 发现的问题 | 红灯信号、信息缺口、薄弱环节 | 生成时重点规避或弥补 |
| 建议 | 改进方向 | 生成时的参考策略 |

同时读取 `data/{用户标识}/internal-resume.md` 和 `data/{用户标识}/profile.md`，获取完整的经历细节和用户画像。

### Step 1.3 确认可用性

向用户确认审阅报告的时效性：

> "我找到了 {日期} 的审阅报告。你的经历有没有在这之后有新的变化？如果没有，我将基于这份报告来生成外部简历。"

如果有新增经历，建议用户先更新审阅报告（运行 resume-analyzer），或在本轮生成时手动补充。

## Phase 2：生成外部简历

**目标：** 为特定目标岗位定制外部简历。

### Step 2.1 明确目标

向用户确认：
> "你想用这份简历投什么岗位/什么类型的公司？如果有多个方向，我们可以分别生成不同版本。"

**需要明确的信息：**
- 目标岗位（如"前端开发工程师"）
- 目标公司类型（如"互联网大厂"、"外企"、"创业公司"，可选）
- 其他特殊要求（可选）

### Step 2.2 信息筛选与重组

**基于审阅报告中的信息提取结果，按目标进行筛选：**

| 操作 | 含义 | 示例 |
|------|------|------|
| **保留** | 与目标岗位强相关的经历，完整保留 | 目标是前端开发，保留前端项目经历 |
| **强化** | 能证明目标岗位核心能力的细节，增加篇幅和细节 | 突出性能优化的具体方案和量化结果 |
| **弱化** | 与目标无关但有一定价值的经历，压缩为一句话 | 非技术类社团经历用一句话带过 |
| **过滤** | 与目标完全无关的"干扰信息"，不出现 | 目标是开发岗，过滤纯行政类经历 |

**注意：** "干扰信息"不是"差的信息"，而是"与当前目标无关的信息"。同一段经历在不同目标下可能从"干扰信息"变成"核心亮点"。

**筛选时参考审阅报告中的建议：**
- 审阅报告中标记为"红灯信号"的内容，在外部简历中需要重点规避或重新表述
- 审阅报告中标记为"被忽略的亮点"的内容，在外部简历中需要主动突出

### Step 2.3 四维对齐

**基于审阅报告中的四维审视结果，确保外部简历在每个维度都有证据：**

| 维度 | 审阅判定 | 对齐策略 |
|------|---------|---------|
| HR 视角 | ✅ 通过 | 保持现有呈现方式 |
| HR 视角 | ⚠️ 薄弱 | 补充匹配度证据（技能关键词、相关经历前置） |
| HR 视角 | ❌ 缺失 | 在开头部分增加与目标岗位相关的概述，用其他维度的证据间接弥补 |
| 技术/专业视角 | ✅ 通过 | 保持现有呈现方式 |
| 技术/专业视角 | ⚠️ 薄弱 | 增加技术细节描述，突出问题解决过程 |
| 技术/专业视角 | ❌ 缺失 | 将非技术经历中的可迁移能力重新表述为专业能力 |
| 业务视角 | ✅ 通过 | 保持现有呈现方式 |
| 业务视角 | ⚠️ 薄弱 | 增加业务场景描述，用业务指标替代纯技术指标 |
| 业务视角 | ❌ 缺失 | 挖掘经历中的业务影响，即使间接也要体现 |
| 决策层视角 | ✅ 通过 | 保持现有呈现方式 |
| 决策层视角 | ⚠️ 薄弱 | 增加主动发起、推动、优化的描述 |
| 决策层视角 | ❌ 缺失 | 挖掘经历中的自驱力证据，重新组织叙述突出成长线 |

**简历优化检查清单：**
- [ ] 每段经历是否有具体的场景描述？
- [ ] 是否有量化结果？
- [ ] 是否体现了个人贡献（区别于团队成果）？
- [ ] 是否有成长线索？
- [ ] HR 视角：6 秒扫描能抓住重点吗？
- [ ] 技术视角：30 秒内能判断专业深度吗？
- [ ] 业务视角：能看到业务场景和理解力吗？
- [ ] 决策层视角：能看到成长潜力和自驱力吗？

### Step 2.4 生成并保存

生成外部简历，保存到 `data/{用户标识}/external-resumes/resume-{目标}.md`。

**外部简历中需要记录的元信息：**
- 目标岗位/方向
- 创建日期
- 版本（正式版 / 预览版）
- 基于的审阅报告日期
- 四维对齐策略（每个维度做了什么调整）
- 与内部简历的差异（保留/过滤/重组了什么）

### Step 2.5 用户审核环节（必须执行）

**生成外部简历后，不能直接交付，必须让用户审核。**

**审核流程：**

> "简历初稿已经生成好了。请你仔细看一下，重点关注以下几个方面：
> 1. **准确性**：有没有写得不准确的地方？（比如夸大了、写错了、遗漏了重要内容）
> 2. **舒适度**：有没有你觉得"不太想这样写"的地方？（简历是你的，你有最终决定权）
> 3. **完整性**：有没有你觉得应该加上但没加的内容？
>
> 你可以逐条提修改意见，我帮你调整。"

**审核原则：**
- 用户说"不准确"→ 必须修改，不能说服用户接受
- 用户说"不太舒服"→ 了解原因，给出专业建议后由用户决定
- 用户说"能不能加上 XX"→ 评估是否相关，相关则加，不相关则解释原因
- 用户说"看起来挺好的"→ 再追问一句"有没有哪个部分你觉得可以更好？"

**审核记录：** 将用户的修改意见和最终决定记录到外部简历的元信息中。

### Step 2.6 多版本简历管理

**同一用户可以有多份外部简历，针对不同目标岗位/方向：**

```
external-resumes/
├── resume-前端开发-互联网大厂.md      # 目标：前端开发，大厂方向
├── resume-前端开发-创业公司.md        # 目标：前端开发，创业公司方向
├── resume-产品经理.md                 # 目标：产品经理（跨方向）
└── resume-全栈开发-v2.md             # 同一目标的更新版本
```

**版本管理规则：**
- 每份简历文件名包含"目标岗位/方向"
- 如果同一目标有多次迭代，用 v2/v3 标注
- 每份简历的元信息中记录"基于的审阅报告日期"和"最后更新日期"
- 面试时应该使用对应目标的外部简历（interview-coach 会自动匹配）

**切换目标时：**
> "你现在想投 [新目标] 的岗位，跟之前的 [旧目标] 方向不同。我建议基于同一份内部简历，为 [新目标] 重新生成一份外部简历。之前的简历不会删除，你可以随时切换。"

### Step 2.7 ATS 优化建议

**ATS（Applicant Tracking System）是大多数公司使用的简历筛选系统。** 外部简历需要兼顾"人读"和"机读"：

**ATS 优化检查清单：**

| 检查项 | 要求 | 常见问题 |
|--------|------|---------|
| **格式简洁** | 不使用复杂排版（表格、图形、多栏） | PDF 转换后格式错乱 |
| **关键词匹配** | 包含目标岗位 JD 中的核心关键词 | 简历用了"同义词"但 ATS 不认 |
| **标准标题** | 经历标题用标准岗位名称（如"前端开发工程师"而非"前端小能手"） | 创意标题 ATS 无法识别 |
| **日期格式统一** | 所有日期用同一格式（YYYY.MM - YYYY.MM） | 格式混乱导致 ATS 解析错误 |
| **无特殊字符** | 避免使用特殊符号、emoji、非标准字体 | 解析时出现乱码 |
| **教育经历完整** | 学校名称、专业、学历、时间齐全 | 缺少毕业时间导致 ATS 无法筛选 |
| **联系方式清晰** | 手机、邮箱在最顶部，格式标准 | 联系方式被解析到错误字段 |

**ATS 优化建议输出：**

> "关于 ATS 兼容性，我检查了你的简历，发现以下需要注意的点：
> - ⚠️ [具体问题 1]：建议修改为 [具体建议]
> - ✅ [做得好的地方]
>
> 这些优化不影响简历的内容，只是让它在系统筛选时不会被误判。"

## Phase 3：回写

**目标：** 如果生成外部简历的过程中发现了新的认知，同步更新内部简历和用户画像。

### 回写触发条件

在生成外部简历的过程中，以下情况需要回写：

| 触发场景 | 回写目标 | 回写内容 |
|---------|---------|---------|
| 重新组织信息时发现了新的亮点 | `internal-resume.md` | 在对应经历下补充新发现的亮点描述 |
| 发现某段经历的能力证据被低估 | `internal-resume.md` | 更新该经历的能力证据标签 |
| 发现用户的成长轨迹比审阅报告记录的更清晰 | `profile.md` | 更新成长轨迹、优势领域 |
| 发现新的可迁移能力 | `profile.md` | 增加能力标签 |

### 回写规范

- **只追加，不删除已有内容**
- 在回写处标注来源：`<!-- resume-optimizer {日期} 回写 -->`
- 如果目标文件不存在，创建新文件
- 回写完成后告知用户更新了哪些内容

## 预览模式

**任何阶段均可触发。** 用户可能只是好奇"以我现在的经历，简历能写成什么样"，或者没有审阅报告但想先看看效果。

**预览模式规则：**
- 不要求有审阅报告，基于 `internal-resume.md` 和 `profile.md` 直接生成
- 如果连内部简历也没有，基于用户口头描述直接生成
- 产出的简历标注为"预览版"
- 明确告知用户哪些地方因为信息不足而显得薄弱
- 预览版的作用是**让用户直观看到差距**，从而更有动力补充经历或运行 resume-analyzer
- 预览版可以随时迭代为正式版（在运行 resume-analyzer 获取审阅报告后重新生成）

**预览模式与正式模式的区别：**

| 对比项 | 正式模式 | 预览模式 |
|--------|---------|---------|
| 审阅报告 | 必须有 | 不要求 |
| 信息筛选依据 | 审阅报告的四维审视 + 信息提取 | 基于经验判断 |
| 四维对齐 | 基于审阅结果精准对齐 | 尽力覆盖，标注不确定处 |
| 产出标注 | 正式版 | 预览版 |
| 回写 | 正常回写 | 正常回写 |

## 下一步衔接

外部简历生成后，可以切换到以下独立 SKILL：

| SKILL | 触发条件 | 传递的输入 |
|-------|---------|-----------|
| **resume-job-matcher** | 用户想进入实际求职（岗位匹配/公司分析） | 外部简历 |
| **interview-coach** | 用户想开始准备面试 | 外部简历 + 经历资产 |
| **resume-analyzer** | 用户想更新审阅报告（新增经历后） | internal-resume.md |

## 职场轨道特别说明

当用户是职业人士时：
- 外部简历侧重"职业叙事"：如何把多年工作经历串成一个有逻辑的故事
- 信息筛选时"管理经历"和"跨部门协作"在多数目标下应保留而非过滤
- 四维对齐增加"转型合理性"的评估（如果用户在转型）

## 与其他 SKILL 的关系

```
career-navigator
  └── Phase 2 经历深潜 ──→ resume-analyzer（信息收集 + 四维审视）
                              │
                              ├──→ resume-optimizer（审阅结果 → 生成外部简历）
                              │       │
                              │       └──→ 回写 internal-resume.md / profile.md
                              │
                              ├──→ resume-job-matcher（审阅结果 → 替代 Part A）
                              └──→ career-navigator（审阅结果 → 更新经历资产）
```

**数据流向：** resume-analyzer 产出审阅报告 → resume-optimizer 读取审阅报告 → 生成外部简历 → 回写新认知到 internal-resume.md 和 profile.md。

## 对话节奏控制

- 每轮最多 3-5 个问题
- 生成外部简历时，先确认目标再开始筛选
- 每轮结束前存档（即使未完成全部生成，也要保存当前进度）
